# APK 多模型 OCR 与包装日期格式矩阵

## 目的与边界

本文记录 Android APK V0.3.11 的本地多模型 OCR 分工、常见包装日期写法、无标签日期推断规则和真实样本验收状态。

识别结果是“待用户核对的录入候选”，不是食品安全结论。任何条码、OCR、日期推断或模型融合结果都只能进入确认页和新增表单，绝不自动保存到食品列表。`expiryDate` 仍是最终可食用日期、排序和提醒的唯一规范字段。

## 协同架构

```text
相机帧 / 视频抽帧 / 图库图片
  -> 原图 + 日期 ROI + 放大裁剪
  -> OpenCV 低对比增强变体
  -> ML Kit Chinese / Latin 通用包装文字 + ML Kit 条码
  -> ONNX Runtime + PP-OCRv6_small_det 文字区域定位
  -> PP-OCRv6_rec_small 对检测区域与日期带复识
  -> 条码与商品信息补充
  -> 商品名门槛 + 日期标签、格式与上下文规则
  -> 字段级跨帧融合、冲突与锁定
  -> 候选确认页
  -> 用户核对或修改
  -> 用户主动保存
```

### 模型分工

| 组件 | 主要职责 | 不负责 |
| --- | --- | --- |
| ML Kit Text Recognition v2 Chinese | 中文包装正文、字段标签、品牌和说明文字的通用召回 | 不单独决定生产日期或有效期 |
| ML Kit Text Recognition v2 Latin | 数字、拉丁品牌、MFG/MFD/EXP/BBE 等混合文本补充 | 不绕过日期规则和确认页 |
| ML Kit Barcode Scanning | 本地条码检测，为商品库查询和商品名优先级提供稳定商品码 | 不把查询结果直接保存 |
| ONNX Runtime Android 1.27.0 | 执行打包在 APK 内的 ONNX 检测和行识别模型 | 不负责字段语义、跨帧投票或保存 |
| `PP-OCRv6_small_det` | 在关键帧中定位中文包装文字、日期喷码和低对比文字区域 | 不直接输出最终字段，不替代用户确认 |
| `PP-OCRv6_rec_small` | 识别已由代码裁剪、缩放和增强的日期文字行，补充点阵字、喷码和同色压印 | 不扫描整幅包装，不承担方向分类或自动保存 |
| OpenCV | CLAHE 局部对比增强、背景差分和裁剪放大，为各 OCR 模型提供互补图像变体 | 增强结果不能作为唯一可信证据 |
| 日期解析器 | 标签关联、日期归一化、保质期换算和受控无标签推断 | 不从任意数字猜测日期 |
| 跨帧投票 | 合并不同模型、不同图像变体和不同视频帧的重复证据，抑制候选跳变 | 不把模型一致误当成用户确认 |

`PP-OCRv6_small_det` 是本地文字区域定位器，`PP-OCRv6_rec_small` 是裁剪文字行复识器；两者与 ML Kit、固定日期 ROI 和 OpenCV 低对比变体互补。最终 APK 不包含 RapidOCR Android AAR、Kotlin 标准库或方向分类模型。后续若更换识别模型，必须用同一批样本重新比较准确率、耗时、发热和 APK 体积，不能只因模型版本更新就直接替换。

当前 `minSdk` 为 24。当前 debug APK 约 220 MB；模型只保留当前链路需要的检测和识别文件，并设置可交付 APK 必须小于 1 GB 的硬门槛。

### 证据优先级

1. 日期附近有明确字段标签，并且日期合法。
2. 同一标准化值被多个模型、图像变体或相邻视频帧重复读到。
3. 日期位于日期 ROI、喷码区或明确标签附近。
4. 无标签日期仅按本文的受控规则生成较低置信候选。
5. 条码、营养表、净含量、产品批号和普通长数字不作为无条件日期证据。

若候选冲突，页面应保留多个候选供用户选择，不应让新帧持续覆盖当前稳定值。用户手动选择或修改后，该字段不再被后台识别覆盖。

## 商品名智能门槛

商品名不是“最长 OCR 行”或“分数最高的任意文字”。页面最多展示 1 个最佳高可信名称，并按以下证据顺序处理：

1. 稳定条码查询命中商品库时，商品库名称优先，包装 OCR 不覆盖它。
2. `产品名称`、`商品名称`、`食品名称`、`品名` 等明确标签后的合法名称获得强证据。
3. 无标签候选必须符合食品名称结构，并达到单帧高分，或在至少 2 帧重复；稳定自动锁定通常要求 3 帧。
4. 食品词、短而完整的名称、画面中的有效文字尺寸和位置可以加权，但不能单独绕过过滤规则。

以下内容不显示为商品名：说明书长句、医疗/药品描述、营养成分与配料、厂家或公司名称、地址电话、生产许可与标准号、营销口号、日期、条码、日文假名或混合脚本噪声及低分内容。最终版本已用最新 APK 单独视觉复测；过滤前截图中出现说明书文字不视为最终版本 PASS。

## 字段标签矩阵

标签允许空格、全角标点和常见 OCR 字符损坏；英文匹配不区分大小写。

### 生产或包装日期

| 类别 | 常见写法 |
| --- | --- |
| 中文生产 | `生产日期`、`生产时间`、`生产日`、`生产批号` |
| 中文包装 | `包装日期`、`加工日期`、`分装日期`、`灌装日期` |
| 中文制造 | `制造日期`、`出厂日期`、`烘焙日期`、`喷码` |
| 英文 | `PROD DATE`、`PRODUCTION DATE`、`PROD`、`MFG`、`MFG DATE`、`MFD` |
| 英文短语 | `DATE OF MANUFACTURE`、`MANUFACTURED ON`、`MADE ON`、`PACKED ON`、`PACKING DATE`、`BOTTLED ON` |

`产品批号`、`LOT`、`BATCH` 本身优先视为批次信息，附近数字不会仅因长得像日期就自动充当日期。只有同时存在可靠生产日期标签或符合专门批次日期规则时，才生成待确认候选。

### 保质期时长

| 类别 | 常见写法 |
| --- | --- |
| 中文 | `保质期`、`保鲜期`、`保存期`、`货架期`、带时长的 `有效期` |
| 储存语境 | `常温 180 天`、`冷藏 7 天`、`冷冻 12 个月` |
| 英文 | `SHELF LIFE`、`VALIDITY`、`STORAGE TIME` |
| 单位 | `天/日`、`周/星期`、`个月/月`、`年`、`day(s)`、`week(s)`、`month(s)`、`year(s)` |

周统一换算为天后再计算 `expiryDate`。没有字段标签的时长只在范围合理、且不处于年龄/月龄等语境时作为弱候选。营养表数值、净含量和数量不能用作保质期。

### 有效期或最终日期

| 类别 | 常见写法 |
| --- | --- |
| 中文有效期 | `有效期至`、`有效日期至`、`有效期限至`、`保质期至` |
| 中文截止 | `失效日期`、`限用日期`、`到期日/到期日期`、`截止日期` |
| 中文食用 | `食用期限`、`使用期限`、`最佳食用期至`、`最佳赏味期`、`此日期前最佳`、`建议食用日期至` |
| 英文缩写 | `EXP`、`EXD`、`BB`、`BBE` |
| 英文短语 | `EXPIRY DATE`、`EXPIRATION DATE`、`BEST BEFORE`、`BEST BY`、`BEST IF USED BY`、`USE BY`、`CONSUME BEFORE` |

“保质期 12 个月”表示时长，“保质期至 2027-09-11”表示最终日期；解析时必须根据“至/到”、日期形态和附近标签区分。

## 日期格式矩阵

| 包装写法 | 示例 | 标准化候选 | 说明 |
| --- | --- | --- | --- |
| 四位年分隔日期 | `2025-03-13`、`2025/3/13`、`2025.03.13` | `2025-03-13` | 支持 `- / .` |
| 中文年月日 | `2025年3月13日` | `2025-03-13` | “日”可缺失 |
| 两位年分隔日期 | `25.03.13`、`25/3/13` | `2025-03-13` | 按 `20YY` 归一化 |
| 八位紧凑日期 | `20250313` | `2025-03-13` | 必须通过合法日期校验 |
| 六位紧凑日期 | `250313` | `2025-03-13` | 按 `YYMMDD` 与 `20YY` 归一化 |
| 年月有效期 | `2029.02`、`2029-2`、`2029年2月` | `2029-02-28` | 仅作为月级有效期候选，取当月最后一天 |
| 紧凑年月有效期 | `202902` | `2029-02-28` | 仅在合法年月与有效期语境下使用 |
| 月/年有效期 | `03/2029` | `2029-03-31` | 四位年在后时按 `MM/YYYY` |
| 双日期区间 | `20250912/20270911` | 生产 `2025-09-12`，有效期 `2027-09-11` | 仅两个日期均合法且先后顺序正确时采用 |

月级有效期按月末只是用于形成可编辑的 `expiryDate` 候选。例如 `2028年2月` 归一化为闰年 `2028-02-29`。确认页必须显示“包装标到月，按月末”之类的提示，让用户知道包装没有提供具体日。

以下格式不做无条件猜测：含三个以上无标签日期、无法区分 `DD/MM/YY` 与 `MM/DD/YY` 的短格式、儒略日/工厂内部编码、只有批号没有日期标签的数字、日期校验不合法的字符串。它们应保留原文或让用户核对，而不是强行填入。

## 无标签推断

日期标签优先级高于以下推断。推断只在合法日期候选数量和上下文满足条件时运行。

| 输入情况 | 候选规则 | 置信边界 |
| --- | --- | --- |
| 恰有两个不同日期，均无标签 | 按时间升序，较早日期作为生产日期，较晚日期作为有效期 | 待确认；不能覆盖更强标签证据 |
| 恰有一个无标签日期，且晚于识别当天 | 作为有效期候选 | 待确认；可能是未来批次计划日期 |
| 恰有一个无标签日期，且等于或早于识别当天 | 作为生产日期候选 | 待确认；不能自动推导保质期 |
| 已有明确生产日期，另有一个更晚的无标签日期 | 更晚日期作为有效期候选 | 仅补齐缺失字段 |
| 已有明确有效期，另有一个更早的无标签日期 | 更早日期作为生产日期候选 | 仅补齐缺失字段 |
| 三个及以上无标签日期或先后关系冲突 | 不自动分配 | 展示原文或多个候选供选择 |

比较基准是设备本地识别当天，而不是随着食品存放时间持续重算。日期候选进入新增表单后仍由用户确认；后续提醒只使用用户最终保存的 `expiryDate`。

## 低对比与激光喷码处理

同色压印、激光去色喷码和浅色纸盒的问题不是简单“调高亮度”即可解决。流水线保留互补变体：

- 原图：保留真实笔画和颜色，避免增强伪影。
- CLAHE：提高局部灰度差，适合轻度低对比和不均匀照明。
- 背景差分：用较大尺度模糊估计背景，再突出比背景略亮或略暗的细线。
- 日期 ROI 放大：减少包装正文和营养表对检测器的干扰，让小字占更多像素。
- 多模型与多帧复核：单个增强图的孤立结果不能直接锁定，重复标准化值才增加权重。

强反光、失焦和运动模糊会真实抹掉笔画。预处理无法恢复不存在的像素，因此 UI 仍应引导用户让日期区域占据画面、保持稳定和改变拍摄角度。

## 多模型变更验收清单

- 本地逻辑测试覆盖标签、日期格式、闰年月末、保质期单位和无标签推断。
- 商品名最多显示 1 个最佳高可信名称，条码库命中优先；说明书、医疗内容、公司名、混合脚本乱码和低分噪声不显示。
- Gradle 构建成功，覆盖安装不清空既有食品数据。
- APK 小于 1 GB，且只包含当前链路需要的关键 OCR 模型。
- 指定视频通过“选择视频 -> 自动识别 -> 候选稳定 -> 填入确认表单”，不手工输入识别字段。
- 指定低对比图片通过“选择图片 -> 增强与多模型识别 -> 日期候选 -> 确认表单”。
- 确认页字段与画面肉眼可见内容一致；月级有效期提示清楚。
- 在用户点击保存前，首页食品数据没有新增。
- 保存后再完成详情、编辑、删除回归；识别失败时仍可退出或重新识别，不写入半成品。
- 为关键页面和样本保留截图或录屏证据，并分别标记自动化、模拟器视觉和真机状态。
