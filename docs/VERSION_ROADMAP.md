# 版本路线图

本文件由 project-architect 维护，用于定义“食期管家”的版本级目标、阶段边界和交给 main-brain 的大任务。

main-brain 负责在每个版本或阶段内部拆解任务、安排小步骤、请求 codex-task-generator 生成 `/goal + require.txt`，并推动 Codex PR、代码审核和文档同步。

## 1. 分工原则

### project-architect 负责

- 定义版本级目标，例如 MVP、第二阶段、后续 AI 阶段。
- 决定每个版本的阶段边界和不可越界事项。
- 判断是否需要补充项目地基，例如文档、权限、工具策略、Skill 规则。
- 当用户想引入新工具、依赖、Skill、外部 API 或开发平台时，负责评估收益、风险、引入时机和是否交给 Codex 执行。
- 给 main-brain 下达版本级或阶段级任务。

### main-brain 负责

- 接收 project-architect 给出的版本级或阶段级任务。
- 细化当前阶段内部的开发顺序。
- 决定当前阶段的第一个、第二个、第三个可执行小任务。
- 要求 codex-task-generator 生成具体 `/goal + require.txt`。
- 跟进 Codex PR、code-reviewer 审核和 documentation-ai 文档更新。
- 阶段完成后写交接文件给 project-architect。

### codex-task-generator 负责

- 把 main-brain 给出的具体小任务转成 Codex 可执行的 `/goal + require.txt`。
- 不负责决定版本路线图。

### Codex 负责

- 根据 `/goal + require.txt` 执行代码或文档修改。
- 提交 commit 并创建 PR。
- 不负责决定阶段目标或版本路线。

## 2. 当前版本：MVP v0.1

版本名称：MVP v0.1 食品库存与保质期基础版

版本目标：完成一个可用的、适合家庭和老年用户的食品库存与保质期管理最小版本。

MVP v0.1 必须支持：

1. 食品日期计算基础。
2. 手动新增食品。
3. 编辑和删除食品。
4. 食品列表展示。
5. 按 `expiryDate` 排序。
6. 到期状态显示。
7. 分类、数量、剩余数量、单位、保存方式和备注。
8. 简单筛选。
9. 简单总览统计。
10. 本地数据或早期云数据库保存。
11. 老年友好基础 UI：按钮清楚、文字直接、流程短。

MVP v0.1 不做：

- 条形码扫描。
- 商品信息查询。
- OCR / AI 信息抽取。
- 大模型对话。
- 菜谱规划。
- 复杂智能提醒。
- 复杂数据看板。
- 家庭成员权限系统。
- 前端密钥存储。

## 3. MVP v0.1 阶段划分

### 阶段 1：核心数据与日期计算基础

目标：建立 `expiryDate` 相关代码基础，为录入、列表、排序、提醒和识别确认流程打底。

阶段范围：

- 日期计算工具函数。
- 支持生产日期 + 保质期计算 `expiryDate`。
- 支持天、月、年单位。
- 明确无效输入处理。
- 明确月底、闰年和日期偏移风险。
- 提供基础验证说明。

不做：

- 页面 UI。
- 食品录入页。
- 食品列表页。
- 云数据库。
- 条形码、OCR、AI、提醒调度。
- 新依赖引入，除非用户确认。

交给 main-brain 的任务：

> 组织完成阶段 1 的日期计算代码基础。由 main-brain 决定阶段内任务顺序，并让 codex-task-generator 生成具体 `/goal + require.txt`。

### 阶段 2：食品基础数据结构与本地 mock 数据

目标：让小程序有可展示和可测试的食品数据结构，先不连接云数据库。

阶段范围：

- 根据 `docs/DATA_MODEL.md` 创建前端可使用的食品数据结构。
- 准备 mock 食品数据。
- 保持 `expiryDate`、`category`、`quantity`、`remainingQuantity`、`unit`、`storageMethod` 等字段一致。
- 为后续列表和录入页面提供稳定数据基础。

不做：

- 云数据库。
- 云函数。
- 条形码、OCR、AI。

交给 main-brain 的任务：

> 在日期计算工具稳定后，组织完成 MVP 食品数据结构和本地 mock 数据基础。

### 阶段 3：食品列表与状态展示

目标：让用户能看到食品库存，并按到期日期理解状态。

阶段范围：

- 食品列表页面或首页列表区域。
- 按 `expiryDate` 排序。
- 无有效 `expiryDate` 的食品排在底部或明确提示。
- 显示已过期、今日到期、即将到期、正常等状态。
- 显示分类、剩余数量、单位、保存方式等基础信息。
- 保持老年友好：文字清楚、卡片信息不过载、主要状态明显。

不做：

- 复杂筛选。
- 复杂统计图表。
- 云数据库。

交给 main-brain 的任务：

> 组织完成食品列表和到期状态展示，确保列表围绕 `expiryDate` 工作。

### 阶段 4：手动新增与编辑删除

目标：让用户可以手动维护食品记录。

阶段范围：

- 新增食品表单。
- 编辑食品。
- 删除食品。
- 支持计算模式和手动 `expiryDate` 模式。
- 支持分类、数量、剩余数量、单位、保存方式、备注。
- 基础输入校验。
- 保存前用户可确认关键信息。

不做：

- OCR 自动填表。
- 条形码自动填表。
- 云端同步冲突处理。

交给 main-brain 的任务：

> 组织完成手动录入、编辑、删除的 MVP 闭环，优先保证流程短、按钮明显、字段不混乱。

### 阶段 5：筛选、总览统计与本地保存

目标：让 MVP 更可用，支持基础管理。

阶段范围：

- 全部、即将过期、已过期、按分类筛选。
- 食品总数、即将过期数量、已过期数量。
- 本地保存或早期云数据库保存，由 main-brain 根据项目状态提出建议，用户确认。
- 保持数据字段与 `docs/DATA_MODEL.md` 一致。

不做：

- 复杂图表。
- 智能提醒。
- AI 分析。

交给 main-brain 的任务：

> 组织完成 MVP 基础管理能力，让用户可以实际试用一轮食品库存管理。

### 阶段 6：MVP 收尾、测试与体验版准备

目标：让 MVP 可演示、可测试、可复盘。

阶段范围：

- 手动测试清单。
- 关键边界检查。
- 老年友好体验检查。
- README 和 docs 同步。
- 阶段复盘交接给 project-architect。

不做：

- 新增大功能。
- 条形码、OCR、AI。

交给 main-brain 的任务：

> 组织完成 MVP v0.1 收尾和测试，为下一版本是否进入云开发、条形码或提醒打基础。

## 4. 第二版本：v0.2 省力录入与提醒基础

进入条件：MVP v0.1 可用，手动录入、列表、排序和保存稳定。

候选目标：

- 云数据库用户隔离。
- 云函数基础。
- 规则化提醒计划。
- 快速修改 `remainingQuantity`。
- 条形码扫描入口和商品信息查询调研。

进入 v0.2 前必须由 project-architect 重新评估：

- 是否接入云开发。
- 是否引入条形码 API。
- 是否需要安全威胁建模。
- 是否需要引入新的 Skill 或工具。
- 是否需要更新数据权限文档。

## 5. 第三版本：v0.3 识别与确认流程

进入条件：v0.2 的云端、安全和基础提醒边界稳定。

候选目标：

- 条形码查询结果确认页。
- OCR 图片识别入口。
- OCR 原文和结构化结果确认流程。
- 识别结果修改后保存。

必须坚持：

- 条形码、OCR、AI 结果不能自动入库。
- 密钥只能放云函数或服务端。
- 用户确认是正式保存前的必经步骤。

## 6. 后续版本：AI 对话与食谱建议

进入条件：基础库存数据稳定，云端权限和隐私边界明确。

候选目标：

- 大模型对话。
- 基于库存的菜谱灵感。
- 食品使用顺序建议。
- AI 辅助提醒计划。

必须坚持：

- 不做医疗、营养诊断或绝对食品安全承诺。
- AI 建议必须是辅助信息。
- 用户可控制哪些食品进入 AI 上下文。

## 7. 工具或依赖引入规则

当用户提出引入某个工具、库、API、Skill 或开发平台时，project-architect 负责先评估：

1. 当前阶段是否真的需要。
2. 是否符合 MVP 优先原则。
3. 是否增加学习成本或维护成本。
4. 是否涉及密钥、账号、外部 API 或隐私数据。
5. 是否应先写工具调研文档。
6. 是否可以交给 Codex 引入。
7. 是否需要 code-reviewer 重点审查。

没有 project-architect 或用户明确批准，Codex 不应自行引入工具、依赖或新 Skill。
