# 版本路线图

本文件由 project-architect 维护，用于定义“食期管家”的版本级目标、版本边界和交给 main-brain 的大任务。

重要分工：

- project-architect 负责定义版本级任务，例如 V0、V1、V2、V3。
- main-brain 负责接收某个版本任务后，在版本内部拆阶段、排顺序、推动 Codex PR 和文档同步。
- codex-task-generator 负责把 main-brain 给出的具体小任务写成 `/goal + require.txt`。
- Codex 负责按 `require.txt` 执行代码或文档修改、提交 commit、创建 PR。

## 1. 项目版本总览

当前路线采用大版本方式：

| 版本 | 名称 | 目标 |
| --- | --- | --- |
| V0 | 手动食品库存 MVP | 做出一个不依赖 OCR / AI / 条形码的可用手动版 |
| V1 | 云端数据与提醒基础 | 把 V0 从本地可用推进到云端保存、用户隔离和基础提醒 |
| V2 | 条形码与省力录入 | 支持扫码和商品信息查询，但必须用户确认后保存 |
| V3 | OCR / AI 识别确认流 | 支持拍照识别包装信息，识别结果进入确认页 |
| V4 | AI 对话与食谱建议 | 基于库存提供对话、使用顺序和食谱灵感 |
| V5 | 发布、优化与家庭协作 | 体验优化、发布准备、数据导出和多成员能力 |

当前正在推进：V0。

当前 V0 内部的第一个小阶段“核心数据与日期计算基础”已经由 main-brain 启动，project-architect 不再干预其内部小任务拆分。

## 2. V0：手动食品库存 MVP

### 2.1 版本目标

V0 的目标是完成一个可实际试用的手动版“食期管家”：

用户可以手动录入食品，查看库存，看到临期 / 过期状态，按最终可食用日期排序，并完成基础筛选、统计和保存。

V0 是整个项目的第一条完整闭环。它不追求智能化，追求可靠、简单、可理解、适合家庭和老年用户。

### 2.2 V0 必须完成

V0 交给 main-brain 的大任务：

> 组织完成“手动食品库存 MVP”完整闭环。main-brain 负责在 V0 内部拆分小阶段和 Codex 任务，project-architect 只在版本边界、工具引入、地基缺失或阶段交接时介入。

V0 必须包含：

1. 食品日期计算基础。
2. 食品数据结构与本地 mock 数据或早期本地存储。
3. 手动新增食品。
4. 编辑食品。
5. 删除食品。
6. 食品列表展示。
7. 按 `expiryDate` 排序。
8. 到期状态显示：已过期、今日到期、即将到期、正常。
9. 分类、数量、剩余数量、单位、保存方式和备注。
10. 简单筛选：全部、即将过期、已过期、正常、按分类。
11. 简单总览统计：食品总数、即将过期数量、已过期数量。
12. 基础保存能力：优先本地保存；是否接云数据库由 main-brain 提出，用户和 project-architect 确认。
13. 老年友好基础 UI：按钮清楚、文字直接、流程短、关键操作明显。
14. V0 手动测试清单和文档同步。

### 2.3 V0 不做

V0 明确不做：

- 条形码扫描。
- 商品信息查询。
- OCR / AI 信息抽取。
- 大模型对话。
- 菜谱规划。
- 复杂智能提醒。
- 复杂数据看板。
- 家庭成员权限系统。
- 自动保存识别结果。
- 前端密钥存储。

### 2.4 V0 交付标准

V0 完成时，用户应该可以：

1. 打开小程序。
2. 新增一条食品记录。
3. 选择或填写分类、数量、剩余数量、单位、保存方式和备注。
4. 通过生产日期 + 保质期得到 `expiryDate`，或手动输入 `expiryDate`。
5. 在列表中看到食品，并按 `expiryDate` 排序。
6. 看出哪些食品过期、今日到期、即将到期或正常。
7. 使用简单筛选。
8. 看到基础统计。
9. 关闭再打开后，数据仍然保留，或至少有明确的当前存储策略说明。

### 2.5 V0 中 project-architect 只在这些情况介入

V0 进行中，main-brain 可以自主拆任务和推进。只有以下情况需要回到 project-architect：

1. 是否从本地保存切换到云数据库。
2. 是否引入 Day.js、WeUI、Vant Weapp 或其他依赖。
3. 是否需要调整数据模型核心字段。
4. 是否提前做条形码、OCR、AI 或提醒。
5. 是否发现现有文档、Skill、权限规则不足。
6. V0 完成，需要规划 V1。

## 3. V1：云端数据与提醒基础

### 3.1 进入条件

进入 V1 前，V0 必须已经能手动完成食品库存管理闭环。

### 3.2 版本目标

把 V0 从“本地可用”推进到“可长期使用”：

- 云数据库保存。
- 用户数据隔离。
- 云函数基础。
- 基础提醒能力。
- 快速更新剩余数量。

### 3.3 V1 必须评估

进入 V1 前，project-architect 必须评估：

1. 是否正式启用微信云开发 / CloudBase。
2. 数据库集合和权限设计。
3. 用户 openid 隔离策略。
4. 是否需要云函数。
5. 是否需要订阅消息。
6. 是否需要安全威胁建模。
7. 是否更新 `docs/DATA_MODEL.md`、`AGENTS.md` 和 Skill。

### 3.4 V1 不做

- OCR / AI。
- 大模型对话。
- 菜谱规划。
- 自动识别自动保存。

## 4. V2：条形码与省力录入

### 4.1 进入条件

进入 V2 前，V1 的云端数据和权限边界应稳定。

### 4.2 版本目标

让用户录入食品更省力：

- 条形码扫描入口。
- 商品信息查询。
- 查询结果确认页。
- 用户确认后保存。
- 查询失败时回到手动录入。

### 4.3 V2 必须坚持

1. 条形码查询结果不能自动入库。
2. 商品名称、品牌、规格、分类等必须展示给用户确认。
3. 商品库 API Key 不得放小程序前端。
4. 外部 API 调用应放云函数或服务端。
5. 查询结果不能代表食品安全判断。

## 5. V3：OCR / AI 识别确认流

### 5.1 进入条件

进入 V3 前，应已经有稳定的数据保存、确认流程和云函数边界。

### 5.2 版本目标

支持拍照识别食品包装信息，但必须让用户确认后保存。

V3 候选能力：

- 拍照或选择图片。
- 云函数处理 OCR。
- OCR 原文展示或保留。
- AI 辅助结构化字段。
- 确认页中可编辑识别结果。
- 用户确认后保存到正式食品记录。

### 5.3 V3 必须坚持

- OCR / AI 结果不能自动保存。
- 密钥不得进入前端。
- 低置信度字段必须提示用户检查。
- 用户确认是正式保存前的必经步骤。

## 6. V4：AI 对话与食谱建议

### 6.1 进入条件

进入 V4 前，库存数据、云端权限、隐私边界和用户确认流程必须稳定。

### 6.2 版本目标

让用户可以询问库存情况，并获得使用顺序、食谱灵感和整理建议。

V4 候选能力：

- 查询剩余食品种类和数量。
- 根据临期食品给使用顺序建议。
- 基于库存给食谱灵感。
- AI 辅助提醒计划建议。

### 6.3 V4 必须坚持

- 不做医疗诊断。
- 不做营养诊断。
- 不做绝对食品安全承诺。
- 用户应能控制哪些食品进入 AI 上下文。
- AI 结果是辅助建议，不是自动决策。

## 7. V5：发布、优化与家庭协作

### 7.1 版本目标

把项目从“可用原型”推进到“更适合真实用户使用”。

候选能力：

- 体验版测试。
- UI 和无障碍优化。
- 家庭成员协作。
- 数据导出。
- 批量管理。
- 更完整的测试清单。
- 发布检查清单。

## 8. 工具或依赖引入规则

当用户提出引入某个工具、库、API、Skill 或开发平台时，project-architect 负责先评估：

1. 当前版本是否真的需要。
2. 是否符合 MVP 优先原则。
3. 是否增加学习成本或维护成本。
4. 是否涉及密钥、账号、外部 API 或隐私数据。
5. 是否应先写工具调研文档。
6. 是否可以交给 Codex 引入。
7. 是否需要 code-reviewer 重点审查。

没有 project-architect 或用户明确批准，Codex 不应自行引入工具、依赖或新 Skill。

## 9. 当前给 main-brain 的大任务

当前交给 main-brain 的大任务是：

> 完成 V0：手动食品库存 MVP。

main-brain 应在 V0 内部自行拆分阶段并推进，但必须遵守 V0 边界。

如果 main-brain 需要向 project-architect 回传信息，只在以下情况创建交接：

1. V0 完成。
2. V0 需要改变边界。
3. V0 需要引入新依赖、云开发、工具或 Skill。
4. V0 发现数据模型或安全原则需要调整。
5. V0 被阻塞，需要项目级决策。
