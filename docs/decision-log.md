# 2026-05-31 使用微信小程序原生开发作为默认方向
## 背景

项目目标是开发微信小程序“食期管家”，核心功能包括食品日期记录、保质期计算、排序、提醒和后续拍照识别。

## 选项
微信小程序原生开发
跨端框架
Web 应用后续再适配小程序
## 决定

默认采用微信小程序原生开发。

## 理由

项目目标明确是微信小程序，原生开发更贴近微信生态，方便后续使用微信云开发、订阅消息、云数据库和云函数。

## 风险

原生小程序知识需要学习，包括 WXML、WXSS、页面生命周期和云开发。

## 后续检查

当项目进入复杂 UI 或多端需求时，再评估是否需要引入框架。

# 2026-06-19 新增 Android APK 本地版

## 背景

用户希望将当前“食期管家”小程序转变为可安装的安卓 App（APK）。

## 选项

直接重写现有小程序
新增原生 Android 工程
引入跨端框架

## 决定

在 `apk/` 目录维护原生 Android 本地版工程，并使用 Android SDK 命令行工具直接构建 debug APK。

## 理由

当前仓库没有 Gradle 工程，本机也没有全局 Gradle。使用 Android SDK 直接构建可以在不删除小程序源码、不引入额外生产依赖的前提下产出 APK，并保留 V0 本地闭环能力。

## 风险

当前 APK 是 debug APK，不包含应用商店发布所需的 release 签名证书。安卓端与小程序端是两套界面实现，后续如果长期维护多端，需要评估共享逻辑或跨端框架。

## 后续检查

如果后续要发布到应用商店，需补充 release 签名、版本号管理、图标、隐私说明和真机兼容性测试。

# 2026-06-20 暂时转为 APK 开发并封存小程序端

## 背景

用户明确要求将 APK 和微信小程序拆成两个文件夹，并表明当前开发方向暂时转为 APK，小程序端封存。

## 选项

继续在仓库根目录混放小程序源码和 APK 工程
仅新增 APK 目录但保留小程序源码在根目录
将 APK 与小程序拆成独立顶层目录

## 决定

将 Android APK 工程放入 `apk/`，将已验收的微信小程序 V0 源码放入 `miniprogram/`。当前主线暂时转为 APK 开发，小程序端封存，除非用户后续明确要求恢复。

## 理由

两个平台目录独立后，开发入口更清晰，也能避免后续 APK 迭代误改小程序端。`miniprogram/` 保留历史可运行源码，方便未来需要时恢复小程序开发。

## 风险

历史文档中仍有小程序阶段规划语境，阅读时应以 `README.md`、`.ai-governance/PROJECT_STATE.md` 和本决策为当前状态准绳。

## 后续检查

后续 APK 任务默认面向 `apk/`。如需重启小程序端，应先解封 `miniprogram/` 并更新项目状态、测试命令和任务边界。
