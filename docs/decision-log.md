# 2026-07-05 APK Gradle 构建基线

## 背景

新增 Excel `.xlsx` 导入导出和实时视频生产日期识别需求后，后续会需要 Fastexcel、CameraX、ML Kit Text Recognition v2 Chinese 等 Maven 依赖。原 `apk/build-apk.ps1` 手工构建适合当前 jar 级依赖，但不适合长期维护现代 Android 依赖图。

## 决定

新增 Gradle Wrapper、根项目 `settings.gradle` / `build.gradle` / `gradle.properties` 和 `:apk:app` Android Gradle 模块。当前使用 Android Gradle Plugin 8.13.0、Gradle 8.13，并通过 `compileSdk` minor API 配置匹配本机 Android SDK Platform 36.1。原 `apk/build-apk.ps1` 继续保留为 fallback。

Gradle debug 构建复用 `apk/debug.keystore`，缺失时自动生成同一套本地 debug keystore；手工脚本也继续使用该 keystore。这样 Gradle APK 可以覆盖安装到手工脚本 APK 上，避免因 debug 签名不一致导致必须卸载应用，从而降低本地数据丢失风险。

## 理由

Gradle 是后续 CameraX、ML Kit 和 Excel 依赖接入的最低成本基线；保留手工脚本可以在 Gradle 或 Maven 网络不可用时继续构建 APK。统一本地 debug 签名是本地 APK 升级验证的前置条件，否则 `adb install -r` 会因为签名不一致失败。

## 后续检查

每次提升 APK 版本时，需要同步 `AndroidManifest.xml` 和 `apk/app/build.gradle` 的 `versionCode` / `versionName`。后续引入新的运行时依赖时，需要更新 `apk/THIRD_PARTY_NOTICES.md`，并继续用覆盖安装和截图验证数据保留与 UI 可用性。

# 2026-07-06 APK 统一识别入口和候选锁定策略

## 背景

当前 APK 把 `扫码识别` 和 `识别包装文字` 拆成两个入口，真实使用时会迫使用户先判断包装上到底该扫条码还是识别文字。视频 OCR POC 也暴露了结果按帧变化、候选不稳定、`使用候选` 经常不可用的问题。

## 决定

后续识别能力改为一个统一的“智能识别”流程：同一页面自动尝试条码识别、商品信息查询和包装文字 OCR，并把商品码、商品名、生产日期、保质期和最终日期放入同一个可编辑结果面板。识别结果仍只作为候选，必须经用户确认或编辑后才能填入新增食品表单并保存。

第一阶段优先采用 CameraX + ML Kit Barcode Scanning + ML Kit Text Recognition v2 Chinese，并保留现有 ZXing 作为 fallback。PaddleOCR / PP-OCRv5 作为第二阶段对比实验，只有在同一批 `video/` 样本上证明 ML Kit 不足后再考虑接入主链路。

## 理由

条码和包装文字本质上都是同一次包装识别的依据，分成两个入口会让 UI、候选状态和保存流程割裂。当前准确率问题不只是正则解析不足，更关键的是缺少帧质量门禁、ROI、字段级投票、候选锁定和 UI 防抖。先统一识别管线和稳定模型，可以直接解决“结果一直变化，无法使用候选”的问题，也为后续 PaddleOCR 对比留下清晰指标。

## 后续检查

实现任务必须包含视频模拟视觉 QA，不能只靠代码推断。新增运行时依赖时要更新 `apk/THIRD_PARTY_NOTICES.md`。最终验收需要覆盖：有条码包装、无条码包装、视频样本、图库图片、低置信度手动修正、确认后才保存。

# 2026-07-14 APK V0.3.6 中文包装与激光喷码识别

## 背景

参考视频暴露出首帧误锁、候选持续跳变、中文包装元数据挤占商品名、识别结果未进入表单，以及浅色纸盒上的激光喷码与背景对比度过低等问题。旧结果页同时堆叠预览、原文和按钮，候选难以选择。

## 决策

继续使用可离线打包的 ML Kit Text Recognition v2，但同时启用 Chinese 与 Latin bundled model。视频帧按全图、宽裁剪、中心细节裁剪和旋转条码区域识别；相邻帧交替使用 OpenCV 4.13 CLAHE 全图增强与中心区域增强。短视频使用更密集采样，长录屏使用受控采样；同一图像变体的条码、中文和拉丁识别并行等待，单个超时不再中断整段视频。商品名按文字框大小、位置、食品词、元数据排除、多帧平均分和有限票数排序，最多展示三个可选候选；首帧不再直接锁定，营销口号不参与自动锁定。

激光喷码中形如 `YYYYMMDD/YYYYMMDD` 的双日期按时间顺序作为“生产日期 / 最终可食用日期”候选，OCR 漏掉中间分隔符时也支持连续 16 位双日期。明确双日期证据达到门槛后优先于更多但无上下文的单日期弱证据；无提示单日期只保留为待确认生产日期，不再同时伪造到期日。所有 OCR 结果仍只预填表单，最终写库必须由用户点击保存。

## 模型选择

PP-OCRv5 Mobile 已纳入对比候选，但当前 Windows PaddleOCR 3.4.1 与 PaddlePaddle 3.3.1 的 CPU 推理存在上游 PIR/oneDNN 兼容错误，且直接迁入 Android 会新增另一套原生推理运行时和模型分发风险。本版采用已在目标 AVD 运行的 ML Kit 双模型加 OpenCV 预处理；后续只有在固定样本基准上证明 PP-OCRv5 明显提升且 Android 离线包可控时再替换主模型。

## 版本与兼容

本次 APK 提升为 `versionName 0.3.6`、`versionCode 9`。OpenCV 原生库只打包 `arm64-v8a` 与 `x86_64`，分别覆盖主流 64 位 Android 手机和当前 AVD。数据 JSON schema 与存储键不变，覆盖升级不得清空既有食品。

最终验收记录见 `docs/qa/2026-07-14-v036-chinese-ocr-and-crud-qa.md`：74 个本地逻辑测试通过，6 个视频样本的可见核心内容通过，video05 激光双日期通过，并完成无手输新增、覆盖安装保留、编辑和删除的视觉验收。

## 决策

提醒设置新增“智能提醒”和“固定日期”。智能提醒在食品新增时按距离 `expiryDate` 或更早开封后建议处理日的剩余天数、保存方式和食品类别生成一次，并把节点快照写入食品 JSON；日期自然流逝不重算，只有类别、保存方式、日期或开封状态变化时重算。高关注和开封后期限增加频次，冷冻及长期保存食品降低频次，提醒基准日当天始终保留。固定日期严格使用用户输入的提前天数并始终以 `expiryDate` 为基准。每日简报只在节点当天纳入食品，避免节点前连续重复通知。

# 2026-07-15 APK V0.3.10 采用轻量组合式本地 OCR

## 背景

ML Kit 中文/拉丁通用模型能覆盖包装正文和标签，但对同色压印、低对比激光喷码和点阵小字仍可能漏字。完整引入第二套检测、方向分类和识别框架会重复打包大量模型与原生运行时。用户接受较大的本地 APK，但仍要求控制模型数量、安装包小于 1 GB，并优先提升中文包装和日期识别的可用性。

## 最终架构

V0.3.10 不再采用 RapidOCR Android AAR/Kotlin 架构，最终组合为：

1. ML Kit Text Recognition v2 Chinese / Latin 负责包装中文、英文、字段标签和通用文字框；ML Kit Barcode Scanning 负责条码，条码商品库命中的商品名优先。
2. OpenCV 对原图和日期 ROI 生成 CLAHE、背景差分等低对比增强变体，同时保留原图，避免增强伪影成为唯一证据。
3. Microsoft ONNX Runtime Android 1.27.0 仅运行 `PP-OCRv6_rec_small.onnx`。该模型只识别已经裁剪好的文字行，不承担文字检测、旋转分类、商品名提取或最终字段决策。
4. 日期解析器处理生产、包装、制造、保质期和有效期标签；跨帧规则融合合并 ML Kit、PP-OCRv6、图像变体和视频帧证据，并抑制候选跳变。
5. 所有识别结果只进入候选确认页，绝不自动写入食品数据库；最终保存仍由用户主动完成。

模型文件只保留当前链路需要的关键项。APK 的 `minSdk` 从 23 调整为 24；最终 debug APK 为 212,506,173 字节，约 212 MB，满足小于 1 GB 的交付门槛。

## 商品名可信门槛

商品名按以下顺序决策：

1. 稳定条码查询命中商品库时优先采用商品库名称，不再让包装 OCR 噪声覆盖。
2. 包装行包含“产品名称/商品名称/食品名称/品名”等明确标签时，标签后的合法名称获得高权重。
3. 无明确标签时，候选必须符合食品名称结构，并达到高分或在至少两个视频帧重复；稳定锁定通常要求 3 帧。
4. 最多展示 1 个最佳高可信名称。说明书句子、医疗/药品描述、营养与配料信息、厂家/公司/地址/电话、营销口号、日期条码、混合脚本乱码和低分内容不显示。

该门槛解决“读取到了文字却把说明书或公司名当商品名”的问题。最终版本已用 5 个既有视频和低对比图片单独视觉复测，过滤前截图不作为最终 PASS 证据。

## 日期规则

常见标签和推断规则记录在 `docs/APK_OCR_DATE_FORMAT_MATRIX.md`。带标签日期优先；无标签且恰有两个合法日期时，较早日期作为生产日期候选、较晚日期作为有效期候选；无标签且只有一个日期时，未来日期作为有效期候选，否则作为生产日期候选。包装只标到月份的有效期按该月最后一天生成可编辑候选，并在确认页提示“包装标到月，按月末”。这些规则只用于录入候选，不构成食品安全判断。

# 2026-07-16 APK V0.3.10 统一关键帧证据与公开样本验证边界

## 背景

上一版组合式 OCR 已能融合 ML Kit、OpenCV 和 PP-OCRv6，但仍需把“哪一帧、哪个区域、哪个模型支持某个字段”统一成可见证据，并对公开样本结果建立不夸大的报告边界。`.local/` 中同时存在批量中间结果、针对性修复重跑和终态截图，不能用一个旧文件代替当前结论。

## 决定

1. 相机、视频和图片继续共用同一识别入口；图库图片在识别前处理 EXIF 旋转和镜像方向。
2. 最多保留 3 个兼顾清晰度、曝光、反光、文字覆盖、OCR 置信、时间差和字段覆盖的关键帧。商品名、生产日期、有效期和条码证据保留来源帧与归一化区域框。
3. ML Kit 中文/拉丁文字与条码负责通用检测；OpenCV 生成互补低对比变体；`PP-OCRv6_rec_small` 只对裁剪日期行和日期带复识，不升级为完整 PaddleOCR 检测框架。
4. 字段置信度是模型、帧质量、语义、跨帧票数和来源数量的启发式融合，只显示“高/中/低”，不称为校准概率。
5. 识别页只返回草稿给可编辑新增表单，不直接调用食品存储；最终保存仍由用户主动完成。

## 日期与商品名门禁

- 支持带 `年/月/日`、`./-`、紧凑数字、英文月份、年/月和月/年等已定义格式；月级有效期按月末生成可编辑候选并提示用户。
- 只有月日而没有年份时安全拒绝，不猜年份。晚于设备当天的值不预选为生产日期，也不得据此计算有效期。
- 商品名优先使用稳定条码商品库或明确标签。品牌短词冲突时不拼接猜测，退回画面可见的食品品类短语。

# 2026-07-17 APK V0.3.11 条码商品档案与独立库存批次

## 背景

原有食品记录同时承担“商品是什么”和“这一批何时生产、何时到期”两类信息。直接给旧记录增加数量会把不同生产日期、保质期、开封状态和提醒计划混在一起；把条码当成唯一商品键又无法覆盖同码不同商品、套装或包装变化。

## 决定

1. 每条 `FoodItem` 仍是一条独立库存批次，继续使用不可复用的 `id`；新增 `productProfileId` 表示可复用商品档案，`barcode` 保存用户确认后的 EAN/UPC/GTIN。
2. 同一条码允许关联多个 `productProfileId`，不建立条码到商品的一对一约束。扫码命中多个档案时必须让用户选择，也可以明确选择“作为不同商品新建”。
3. 补货始终建立新批次。只复用名称、分类、单位、保存方式、位置、条码和商品档案 ID；生产日期、保质期、最终日期、数量、开封/用完状态、备注和提醒快照必须重新录入或由本次识别产生。
4. 已知条码再次进入新增流程时先展示本机商品档案，并把主要行为改为“补货新批次”；仍保留“作为新商品”，避免同码不同商品被强行合并。
5. 用户在编辑或补货时改变名称、分类、单位、保存方式、位置或规范化条码，当前批次必须脱离原档案并生成新 `productProfileId`，不能污染同档案的其他批次。
6. `ProductProfileIndex` 只用于从批次构建本地扫码索引；每个档案使用最新批次作为展示代表，但不删除、不合并历史批次。
7. Excel 在原有列后追加 `productProfileId` 与 `barcode`。覆盖导入保留工作簿内关系；追加导入仅在档案 ID、规范化 GTIN 和稳定身份一致时复用本机档案，真实冲突必须保持独立或明确报错。
8. 本地数据 schema 升为 v2。v1 迁移为每条旧记录生成确定性档案 ID，只迁移格式明确且校验位合法的旧备注条码，不按名称或条码擅自合并历史记录；迁移前继续保留备份。

## 安全边界

- 扫码、OCR、商品库查询和日期推断只生成候选，不能直接保存。
- `expiryDate` 继续是排序和提醒的唯一规范最终日期。
- 本地 APK 不增加云存储、账号或密钥；升级和 Excel 往返必须保持批次数据完整。

# 2026-07-19 APK V0.3.14 指定视频保质期与 JSON 快速换机

指定录屏中“保质期：9个月”位于完整包装区域，但原流程只把补充区域的日期候选合并回聚焦结果，遗漏了保质期候选。聚焦解析现在同时合并补充区域的日期与保质期证据；录屏回放按战略时间点选取关键帧，并限制昂贵的 PP-OCR 检测次数。该视频在模拟器中得到生产日期 `2026-01-20`、保质期 `9个月` 和最终日期 `2026-10-20`，仍须由用户在确认页核对后保存。

本地数据新增版本化 JSON 快速换机格式。库存批次 ID 是设备间合并键；schema v3 会为旧数据缺失或重复的批次 ID 做确定性修复。导出可按首次入库日期、生产日期和最终日期的闭区间筛选，文件可保存或交给 Android 系统分享面板。导入先完整预览并校验，文件有错误时不写入；相同 ID 的较新批次会更新完整批次状态和数量，相同内容不重复，较旧文件保留本机数据。该能力不引入账号、服务器或自动云同步。

本次 APK 提升为 `versionName 0.3.14`、`versionCode 17`。JSON 同批次更新、重复导入、重启保留、系统分享面板、Excel 往返预览以及新增、编辑、减少、补货新批次和删除均完成模拟器视觉验收。实际 QQ/微信接收因 AVD 未安装对应应用而未覆盖；真实手机相机仍不在本轮通过范围。

已用完记录的恢复操作统一显示为“恢复在库”，确认框明确说明剩余数量为 0 时会恢复为 1，并重新参与到期提醒，不再使用容易被理解成只打开通知的“恢复提醒”。

## 原因

自动化测试不能代替包装文字、关键帧和页面布局的视觉检查；同时，只有识别样本通过而没有导入导出、误操作恢复和覆盖安装证据，仍不足以交付长期本地使用的 APK。恢复操作会同时修改归档状态、剩余数量和提醒状态，名称应准确表达整体行为。

## 验证

最终证据记录在 `docs/qa/2026-07-18-v0311-final-qa.md`。本轮范围结论为 `PASS`；真实手机相机和更大统计样本集仍为 `PARTIAL`，不得扩大结论。
## 2026-07-18：相机日期专用模式与补货快捷路径

用户反馈近距离对准保质期仍无法识别、补货没有相机输入、卡片外层缺少“减少 1”。本轮将相机分析目标从 `1280x960` 提升为 `1920x1080`，Paddle 文字检测输入从 `640/960` 提升为 `960/1280`，扩大候选文字区域和低对比增强裁剪。补货入口启用日期专用模式：不要求同时识别商品名或条码才完成日期识别，并为低对比日期保留更密集但有上限的重识别机会。相机默认使用轻微变焦、中央日期区域对焦和 8 秒 AF/AE/AWB 测光。

补货表单的日期区新增“相机识别本批次日期”。结果只回填本次补货的生产日期、保质期和有效期，不覆盖复用的商品档案，也不修改旧批次。食品卡片恢复外层“减少 1”，低频归零、编辑和删除仍保留在更多操作中。视觉与自动化证据见 `docs/qa/2026-07-18-camera-replenishment-quick-actions-qa.md`。

## 2026-07-19：V0.3.15 相机可见反馈、对焦恢复与低对比日期协同

相机页原有 CameraX 点按对焦，但初始测光在预览真正建立前立即发起，结果只写在下方结果区；用户无法从取景框确认相机是否收到画面、是否清晰或是否仍在识别。PP-OCR 模型预热还与实时识别共用单线程队列，可能阻塞首帧中文 ML Kit 快速路径。

本轮把模型预热移到独立执行器，实时识别继续只保留最新待处理帧。相机绑定后延迟执行中央 AF/AE/AWB 测光，点按取景框会显示对焦框及“对焦中/完成/失败”状态，3 秒后自动取消一次性测光并回到连续相机控制。取景框内新增两行紧凑状态，分别显示焦点状态，以及画面已收到、清晰度、亮度/反光、中文 OCR 和分析帧数；结果区高度收紧，扫描阶段保留更多可见画面。

低对比日期只在清晰度、曝光和反光达到最低条件时进入最多 3 次恢复。OpenCV 使用 CLAHE、背景差分，以及同时提取暗笔画和亮浮雕边缘的 black-hat/top-hat 形态学变换；PP-OCRv6 对中央日期窄行进行多预处理复识。无标签 `yyyyMM` 残片不再作为生产日期，单次低置信完整误读也必须由高置信或至少两个预处理结果一致才能进入候选。黑色瓶底样本最终正确得到 `2025-09-12` 与 `2027-03-11`。

旧视频回归还暴露了完成阶段的一个边界：完整转录已经读到“保质期 9 个月”和多次 `20260120S7420`，但一次未来噪声 `20360120` 会让生产日期被清空。现在仅在存在保质期时，从过去十年内重复至少两次且唯一领先的完整八位日期恢复生产日期；指定绿色包装视频重新得到生产日期 `2026-01-20`、保质期 `9个月` 和到期日 `2026-10-20`。

实现依据为 Android CameraX `FocusMeteringAction` 与 `ImageAnalysis.STRATEGY_KEEP_ONLY_LATEST`、ML Kit 中文 Text Recognition v2，以及 OpenCV CLAHE 和形态学转换官方文档。识别结果仍只进入核对页，不自动保存。模拟器能验证页面、视频/图片路径和候选结果，但其虚拟相机在 CameraX 启动后会阻塞 `adb screencap`，因此真实手机镜头的自动对焦速度、连续曝光和帧率结论仍为 `PARTIAL`。
