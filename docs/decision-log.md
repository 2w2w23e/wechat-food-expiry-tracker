# 2026-05-31 使用微信小程序原生开发作为默认方向
## 背景

项目目标是开发微信小程序“食期管家”，核心功能包括食品日期记录、保质期计算、排序、提醒和后续拍照识别。

## 选项
微信小程序原生开发
跨端框架
Web 应用后续再适配小程序
## 决定

默认采用微信小程序原生开发。

## 理由

项目目标明确是微信小程序，原生开发更贴近微信生态，方便后续使用微信云开发、订阅消息、云数据库和云函数。

## 风险

原生小程序知识需要学习，包括 WXML、WXSS、页面生命周期和云开发。

## 后续检查

当项目进入复杂 UI 或多端需求时，再评估是否需要引入框架。

# 2026-06-19 新增 Android APK 本地版

## 背景

用户希望将当前“食期管家”小程序转变为可安装的安卓 App（APK）。

## 选项

直接重写现有小程序
新增原生 Android 工程
引入跨端框架

## 决定

在 `apk/` 目录维护原生 Android 本地版工程，并使用 Android SDK 命令行工具直接构建 debug APK。

## 理由

当前仓库没有 Gradle 工程，本机也没有全局 Gradle。使用 Android SDK 直接构建可以在不删除小程序源码、不引入额外生产依赖的前提下产出 APK，并保留 V0 本地闭环能力。

## 风险

当前 APK 是 debug APK，不包含应用商店发布所需的 release 签名证书。安卓端与小程序端是两套界面实现，后续如果长期维护多端，需要评估共享逻辑或跨端框架。

## 后续检查

如果后续要发布到应用商店，需补充 release 签名、版本号管理、图标、隐私说明和真机兼容性测试。

# 2026-06-20 暂时转为 APK 开发并封存小程序端

## 背景

用户明确要求将 APK 和微信小程序拆成两个文件夹，并表明当前开发方向暂时转为 APK，小程序端封存。

## 选项

继续在仓库根目录混放小程序源码和 APK 工程
仅新增 APK 目录但保留小程序源码在根目录
将 APK 与小程序拆成独立顶层目录

## 决定

将 Android APK 工程放入 `apk/`，将已验收的微信小程序 V0 源码放入 `miniprogram/`。当前主线暂时转为 APK 开发，小程序端封存，除非用户后续明确要求恢复。

## 理由

两个平台目录独立后，开发入口更清晰，也能避免后续 APK 迭代误改小程序端。`miniprogram/` 保留历史可运行源码，方便未来需要时恢复小程序开发。

## 风险

历史文档中仍有小程序阶段规划语境，阅读时应以 `README.md`、`.ai-governance/PROJECT_STATE.md` 和本决策为当前状态准绳。

## 后续检查

后续 APK 任务默认面向 `apk/`。如需重启小程序端，应先解封 `miniprogram/` 并更新项目状态、测试命令和任务边界。

# 2026-06-22 APK 扫码识别升级和版本号规则

## 背景

用户要求 APK 的扫码识别更接近正式产品：每次生成 APK 后同步提升版本号；扫码页需要更简洁美观；识别需要覆盖一维商品条码以及中国物品编码/GS1 二维码中的商品码，并提升速度和准确率。

## 决定

本次 APK 提升为 `versionName 0.3.0`、`versionCode 3`。扫码继续使用现有 ZXing core，不引入 OpenCV 微信二维码库；新增对 QR Code / Data Matrix 原始内容的商品码提取，支持 GS1 Digital Link `/01/{GTIN}`、常见 `gtin/code/barcode` 参数和 GS1 AI `(01)` 元素串。扫码页增加补光、动态扫描线、清晰取景框、相册入口和手动输入兜底。

## 理由

当前目标是商品条码和商品信息查询。OpenCV 微信识别库更偏二维码识别，不直接解决 EAN/UPC/GTIN 商品条码查询，并会引入 native 依赖、模型文件和更大的 APK 体积。ZXing 已在项目内使用，继续扩展格式、方向修正、取景框区域优先识别和二维码内容解析，改动更小也更符合当前 APK 本地版边界。

## 后续检查

每次生成可交付 APK 时都要同步递增 `AndroidManifest.xml` 的 `versionCode` 并更新 `versionName`。扫码能力仍需在真实 Android 手机上用反光包装、模糊图片、远距离条码、GS1 Digital Link 二维码和普通商品一维码做兼容性测试。

# 2026-06-22 APK V0.3.1 扫码缩放与进口商品查询增强

## 背景

用户反馈扫码页无法手动缩放，进口商品条码在微信/中国物品编码相关查询中可识别，但 APK 内现有查询源无法识别，且扫码页体验和识别速度需要继续优化。

## 决定

本次 APK 提升为 `versionName 0.3.1`、`versionCode 4`。扫码页新增右侧 `+ / -` 倍率按钮、倍率显示和双指缩放，实时识别节流从 150ms 调整为 90ms。商品信息查询继续先使用现有 GS1 查询接口；当未命中且条码不是国内 `690-699` 前缀商品时，补齐 GTIN-14 后追加查询中国商品信息服务平台公开进口商品接口 `GetImportProductDataForGtin`。

## 理由

中国商品信息服务平台页面本身使用公开接口查询进口商品列表，适合作为进口商品的官方补充数据源。但该平台会对频繁或未登录访问返回限流/浏览范围提示，因此 APK 只做合规查询和用户提示，不内置账号、密钥或绕过限制逻辑。扫码体验优先补上手动缩放，因为远距离、小条码、反光包装和弧面包装通常需要用户把条码放大到更可读的尺寸。

## 后续检查

需要在真机上用远距离一维码、反光包装、弧面包装、进口 EAN-13、GS1 Digital Link 二维码和图库图片继续测试。若后续要达到接近微信的识别稳定性，应评估 CameraX + ML Kit 或服务端商品数据代理，但密钥和账号仍不能写入 APK 客户端。

# 2026-07-05 文档同步 APK 本地主线和本地数据优先级

## 背景

APK 本地版已包含商品码识别、本地提醒、系统通知和已用完归档，但部分 README 和数据模型文档仍把条码、提醒和发布/云端能力写成当前或近期待办，容易误导后续任务调度。

## 决定

当前主线只按 Android 本地 APK 维护文档口径。当前不考虑应用商店发布、云数据库、账号体系或多设备同步；已支持的商品码识别、商品信息查询、本地提醒、系统通知和已用完归档只在本地 APK 边界内维护。后续最高优先级是本地数据 schema 升级、迁移、备份、导出导入和旧数据回归不丢失。

## 理由

APK 已进入本地功能累积阶段，继续把条码和提醒写成“不做”会与现状冲突；过早写入云端或发布计划会扩大当前任务范围。先保证本地数据升级可靠，才能降低后续功能迭代造成数据丢失的风险。

## 后续检查

APK 构建仍依赖本机 Android SDK 命令行环境；如果当前机器缺少 SDK，应将构建验证标记为 BLOCKED，并在具备环境后再验证。后续文档变更需要继续区分“已支持的本地能力”和“暂缓的云端/发布能力”。
