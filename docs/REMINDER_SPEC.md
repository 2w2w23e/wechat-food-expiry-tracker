# Android 本地提醒规则

## 目标与边界

提醒只帮助用户安排查看和处理食品，不判断食品是否绝对安全。`expiryDate` 仍是最终可食用日期和提醒计算的唯一到期基准。智能模式可在有效的“开封后建议处理日”早于 `expiryDate` 时优先使用该日期，但不会覆盖 `expiryDate`；固定模式始终以 `expiryDate` 为基准。

## 用户模式

### 智能提醒

智能模式根据以下本地数据生成提醒日期：

1. 距离有效提醒基准日的剩余天数。
2. 保存方式：冷藏、冷冻、常温、阴凉干燥、避光或其他。
3. 食品类别：肉蛋水产、熟食、乳制品、蔬果、主食、饮料、零食、调味品等。
4. 是否存在更早的开封后建议处理日。

算法完全在本机运行，不调用云端模型。生成结果去重、按提前天数倒序排列，最多保留 6 个节点，并始终保留提醒基准日当天。智能节点不会早于计划生成日，也不会超过生成时的剩余天数。

### 计划生成与持久化

- 新增食品时生成一次智能提醒计划，并把提醒节点、生成日期、生成时剩余天数和输入指纹保存到该食品的本地 JSON 中。
- 日期自然流逝不会重新分档，也不会改变已经生成的提醒日期。
- 修改会影响提醒的输入后重新生成：类别、保存方式、生产日期、保质期、`expiryDate`、开封日期或开封后期限。
- 修改名称、备注、存放位置、数量、剩余数量、已用完状态不会重新生成提醒日期。
- V0.3.7 旧食品或 Excel 导入食品没有提醒快照时，首次载入后生成一次并回写；后续仍遵守上述规则。

### 固定日期

固定模式严格使用用户填写的提前天数，例如 `7,3,1,0`。`0` 表示 `expiryDate` 当天，不会自动切换到开封后建议处理日，也不会按类别或保存方式增减日期。负数、重复值和超过食品完整保质期的节点不会进入计划。

## 智能分级

| 关注级别 | 主要条件 | 调整方向 |
| --- | --- | --- |
| 高 | 肉蛋水产、熟食、冷藏乳制品、短保质期冷藏其他食品，或更早的开封后期限 | 节点更密集，短周期可包含 5、3、2、1、0 天 |
| 中 | 蔬果、短保质期主食、冷藏饮料、短保质期零食，以及一般冷藏食品 | 使用标准密度；冷藏中等级额外补充 2 天节点 |
| 低 | 冷冻食品、长保质期调味品/零食/主食、阴凉干燥或避光长期保存食品 | 减少提醒频次，保留关键节点和到期当天 |

关注级别只用于提醒频率，不是食品安全等级。

## 剩余时长节点

下面是主要基线。具体结果还会经过关注级别、冷藏和开封状态调整。

| 剩余天数 | 高关注 | 中关注 | 低关注 |
| --- | --- | --- | --- |
| 0-3 天 | 2、1、0 | 1、0 | 0 |
| 4-7 天 | 5、3、2、1、0 | 3、1、0 | 0 |
| 8-14 天 | 7、5、3、1、0 | 7、3、1、0 | 7、0 |
| 15-30 天 | 14、7、3、1、0 | 14、7、3、1、0 | 7、0 |
| 31-90 天 | 30、14、7、3、1、0 | 30、14、7、3、0 | 30、7、0 |
| 91 天以上 | 逐级使用 60/90、30、14/7、1/0 | 逐级使用 60/90/180、30、14/7、0 | 逐级使用 90/180、30/60、0 |

表中的档位只在计划生成或相关状态变化时选择一次。之后即使剩余天数进入另一个档位，也继续使用已保存的原提醒节点。

## 到期当天时段

到期当天仍沿用风险时段：高关注默认 `08:30、12:30、18:30`，中关注默认 `09:00、18:00`，低关注默认 `09:00`。用户自定义到期当天时段后，两种模式都使用自定义值。

每天的本地简报只纳入“提醒节点恰好是今天”的食品，不会在某个节点到来前连续多日重复通知。智能模式可额外提示前一天已过期的食品；固定模式不自动增加用户未选择的过期后日期。

## 兼容

V0.3.7 及更早版本没有提醒模式字段。升级到 V0.3.8 后，使用默认 `7,3,1,0` 的旧设置按原有风险规则加载为智能模式；使用自定义提前天数的旧设置加载为固定模式，避免改变原来的提醒日期。原有开关、提前天数和到期当天时段继续保留。食品 JSON 增加可选的智能提醒快照字段，旧数据可直接读取，原存储键和备份机制不变。
