# 功能范围说明

## 1. 产品定位

食期管家新的产品定位是：面向家庭 / 老年友好的食品库存与保质期管理小程序。

产品重点从单一的食品保质期记录，扩展为轻量的家庭食品库存管理。小程序应帮助用户记录食品、查看库存、关注临期和过期状态，并为后续条码识别、商品信息查询、规则化智能提醒、AI 对话和菜谱规划预留扩展空间。

体验上优先照顾家庭和老年用户：流程要短，文字要清楚，主要操作要明显，避免把关键动作藏在复杂菜单里。

## 2. MVP 范围

MVP 只实现最基础、最可靠的食品库存和保质期管理能力。

### 2.1 食品录入

- 支持手动新增食品。
- 支持输入食品名称。
- 支持选择或填写 `category`。
- 支持填写品牌、备注、保存方式等基础信息。
- 支持填写 `quantity` 和 `remainingQuantity`。
- 支持填写单位，例如袋、瓶、盒、个、克、千克、毫升等。

### 2.2 日期与保质期

- 支持通过生产日期 + 保质期计算最终可食用日期。
- 计算结果统一保存到 `expiryDate`。
- 支持用户直接手动输入最终可食用日期。
- 手动输入结果同样保存到 `expiryDate`。
- `expiryDate` 是列表排序和提醒逻辑的唯一标准字段。
- 使用 `dateSource` 标记日期来源，例如 `calculated` 或 `manual`。

### 2.3 列表、排序和筛选

- 食品列表默认按 `expiryDate` 排序。
- 无有效 `expiryDate` 的食品应排在列表底部。
- 支持简单筛选：
  - 状态筛选支持多选，默认选择即将过期，也可不选状态；不选状态时列表不展示食品。
  - 状态筛选范围包括即将过期、已过期、正常。
  - 分类筛选支持多选，不在分类列表中展示“全部分类”选项，改用独立的一键全选和一键重置操作。
  - 分类弹窗存在搜索条件时，一键全选和一键重置只作用于当前搜索结果。
  - 已选筛选项应在当前筛选栏展示，并支持从当前筛选栏移除选择。
  - 当前筛选栏滚动吸顶后应支持收起；回到顶部使用右下角独立箭头按钮。

### 2.4 概览统计

MVP 只做简单统计，不做复杂数据分析。

- 食品总数
- 即将过期数量
- 已过期数量

### 2.5 老年友好 UI 原则

- 使用大按钮，便于点击。
- 使用清楚、直接、短句式的文字。
- 关键流程尽量短，例如新增食品不应被拆成过多步骤。
- 主要操作要可见，例如新增、保存、取消、筛选。
- 避免把关键动作藏在长按、滑动、二级菜单或难发现的入口里。
- 错误提示和空状态说明要明确，不使用晦涩术语。

## 3. MVP 预留但不实现

以下字段或能力可以在数据模型中预留，但 MVP 不实现真实功能：

- `barcode`：条码字段，仅预留。
- `barcodeLookupStatus`：条码查询状态，仅预留。
- `productNameSource`：商品名称来源，例如手动输入、条码查询、OCR 或 AI。
- `reminderPlan`：提醒计划字段，为后续规则化智能提醒预留。
- AI 对话相关字段，例如是否在 AI 聊天中可见、菜谱标签等，仅在需要时预留。

MVP 不调用真实条码 API，不执行商品库查询，不实现 AI 对话，不生成菜谱，不实现 OCR 自动识别或自动保存。

## 4. 第二阶段范围

第二阶段在 MVP 稳定后再实现，重点是让库存信息录入和提醒更省力。

- 条码扫描。
- 商品信息查询。
- 条码查询结果保存前必须由用户确认和编辑。
- 基于规则的智能提醒算法。
- 快速更新 `remainingQuantity`，例如一键减少、标记吃完、快速补货。
- 更清晰的临期、过期、库存不足提示。

第二阶段仍然不应把条码、OCR 或 AI 的识别结果直接写入食品数据。任何外部识别或查询结果都必须经过用户确认后保存。

## 5. 后续阶段范围

后续阶段可以探索更智能的家庭食品管理能力，但不得影响 MVP 的稳定性。

- AI 聊天助手。
- 基于库存的菜谱规划。
- AI 辅助提醒计划。
- OCR / AI 增强信息提取。
- 更完整的家庭协作或多成员场景。

AI 聊天助手只能提供库存整理、菜谱灵感、使用顺序建议等辅助信息，不应做医疗、营养诊断或绝对食品安全承诺。

## 6. MVP 明确不做

MVP 阶段明确不实现：

- 真实条码 API 集成。
- AI 聊天。
- 菜谱生成。
- OCR 自动保存。
- 条码查询结果自动保存。
- AI 识别结果自动保存。
- 复杂数据分析看板。
- 医疗、营养或绝对食品安全判断。
- 复杂家庭成员权限系统。
- 云端密钥管理之外的任何前端密钥存储。

## 7. 安全与隐私原则

- 条码查询结果必须展示给用户确认，用户确认后才能保存。
- OCR / AI 结果必须展示给用户确认，用户确认后才能保存。
- API Key、OCR Key、AI Key、云开发密钥或其他敏感凭据不得放入小程序前端代码。
- 需要密钥的能力应放在云函数或服务端。
- AI 聊天不得做医疗、营养或绝对食品安全声明。
- 食品状态提示只能用于库存和日期管理，不代表食品一定安全或一定不可食用。

## 8. 术语约定

- `expiryDate`：最终可食用日期，是排序和提醒逻辑的标准字段。
- `quantity`：初始或登记数量。
- `remainingQuantity`：当前剩余数量。
- `barcode`：食品条码，MVP 只预留。
- `category`：食品分类，用于展示和筛选。
- `reminderPlan`：提醒计划，MVP 只预留或保存简单配置。
- `dateSource`：`expiryDate` 的来源，例如计算得到或手动输入。
